# Diagnosing a Sparra run

Sparra debugging is *reading the filesystem*. The whole run is on disk under `.sparra/`;
nothing important is hidden in memory. Work from the symptom to the artifact that explains
it, then to the fix.

- [Read the artifacts in order](#read-the-artifacts-in-order)
- [Item statuses](#item-statuses)
- [Failure signatures → cause → fix](#failure-signatures)
- [Clean re-runs](#clean-re-runs)

## Read the artifacts in order

From the project root (the dir you ran `sparra` in):

1. **`.sparra/state.json`** — the phase machine + per-item state. Start here.
   ```bash
   node -e "const s=require('./.sparra/state.json');console.log('phase:',s.phase,'mode:',s.mode);console.log('workspace:',s.build.workspaceNote||'-');if(s.build.waitingUntil)console.log('PAUSED on limit until',new Date(s.build.waitingUntil).toLocaleString());if(s.build.limitedRoles&&Object.keys(s.build.limitedRoles).length)console.log('limited backends:',s.build.limitedRoles);for(const[k,v]of Object.entries(s.build.items||{}))console.log(' ',k,JSON.stringify({status:v.status,round:v.round,pivots:v.pivots,lastScore:v.lastScore,cost:+(v.costUsd||0).toFixed(3),tok:v.tokensUsed}))"
   ```
   `costUsd` is `$0` for Codex roles (it reports tokens) — look at `tokensUsed`. `waitingUntil`/`limitedRoles` appear when `build.autoRestart` paused the run on a provider limit (see signatures). `build.unitWorktrees` (name→`{dir,branch,src}`) is the registry of **persistent per-unit generator worktrees** (`run_role`'s `unitWorktree`); a stale entry whose `dir` is gone means a teardown was interrupted — dispose it with `sparra role rm-worktree --name <name> [--force]`. The inverse (a live worktree whose registry entry a concurrent writer dropped — the registry is a full-file state.json read-modify-write) now **self-heals**: `ensureUnitWorktree` reconciles from `git worktree list` and re-adopts a dir that is a live worktree on exactly `sparra/<name>`, so the old "target path already exists and is not a registered unit worktree" false-refusal no longer fires for your own tree.

2. **`.sparra/workitems/items.json`** — the decomposition. Check the *shape*: how many items,
   and whether any are setup/verification-only (a smell — see signatures). Each item carries
   `id`/`title`/`summary`/`dependsOn`/`rationale` and the optional `gen` (hybrid routing) and
   `relevantPaths` (files whose CODEBASE_MAP sections the generator/contract prefer over the
   head-slice) — both human-editable here before building.
   ```bash
   node -e "const a=require('./.sparra/workitems/items.json');console.log('count',a.length);a.forEach(i=>console.log(' -',i.id,'|',i.title,'| deps',JSON.stringify(i.dependsOn),'| paths',JSON.stringify(i.relevantPaths||[])))"
   ```

3. **`.sparra/contracts/<id>.contract.md`** — the full generator↔evaluator negotiation for
   an item, with the agreed (or force-converged) contract at the bottom (`## AGREED CONTRACT`).
   Count rounds and whether it actually agreed:
   ```bash
   grep -c '^### Round' .sparra/contracts/<id>.contract.md   # 2 lines per round (proposal+critique)
   grep -c 'CONTRACT: AGREED' .sparra/contracts/<id>.contract.md
   ```
   Read the **last critique** to see what the evaluator was unhappy with — substantive
   critiques (verification holes, gameable assertions) are healthy; nitpicking toolchain
   trivia is not (and the prompt is meant to prevent it).

4. **`.sparra/verdicts/<run>/<id>.r<n>.verdict.md`** — per-round grade: weighted total vs
   threshold, per-criterion scores, **failed assertions with evidence**, blocking issues, and a
   `<details>` block with the raw evaluator output. This tells you *why* an item failed a round.
   Autonomous build verdicts live in a **run-scoped** subdir (so runs reusing item ids never
   collide); interactive/loop evaluator role-runs auto-persist a redacted verdict to
   **`.sparra/verdicts/role-run-evaluator-<stamp>.verdict.md`** even with no `out` given.

   **`.sparra/reviews/<id>.r<n>.review.md`** — if `review.enabled`, the code-review findings
   (blocking vs advisory, with `blockOn`). An item that passes the exercise but isn't accepted
   was blocked here.

5. **`.sparra/traces/<run>/NN-<role>.md` / `.sparra/traces/role-run-*/NN-role-run-*.md`** —
   full role transcripts as markdown. The most recent file (by mtime) shows current activity.
   Read these when the verdict/contract summary isn't enough — e.g. to see what the generator
   actually did, or how a non-evaluator exercised the artifact. Evaluator role-run traces may
   contain holdout text, so prefer verdicts. `ls -lt .sparra/traces/*/ | head`.

   **`.sparra/measure/`** — if `measure.enabled`, the post-accept QA step's `baseline.json`
   (metrics keyed by name) + a rendered `measure-<stamp>.md` report per accept (metrics vs.
   baseline, regressions flagged). A `MEASURE` line in `memory.md` summarizes each run. Reflect
   reads this dir, so measured regressions inform prompt improvement.

6. **`.sparra/environment.md`** — optional environment notes injected into writer prompts. Read it
   when tools fail before project code loads (cache/user dirs, headless app setup, writable HOME).

7. **`.sparra/memory.md`** — durable cross-run learnings (pivots, budget halts, pass/fail, measure).
   Roles read this each item; a wrong/misleading entry can bias new work.

8. **`CHANGELOG.md`** / **`.sparra/proposals/`** — recorded deviations (in-scope) and
   out-of-scope ideas logged for the human (brownfield).

## Item statuses

`pending` → `contracting` → `building` → `passed` | `failed` | `budget_exceeded` | `abandoned`.
A run that "died" usually leaves an item stuck at `contracting` or `building` with no verdict —
check the latest trace and the contract file for an error (often a session-level failure).

## Failure signatures

| Symptom | Likely cause | Fix |
|---|---|---|
| **Codex MCP `run_role` dies after about 60 seconds during a multi-minute role** | The server kept Codex's 60-second default `tool_timeout_sec` | Set `tool_timeout_sec` to at least `1800` for the `sparra-run` MCP server, then reload/reinstall the plugin and start a fresh thread. The packaged `.codex-plugin/mcp.json` already declares `1800`. |
| **Headless `codex exec` rejects direct `run_role` as a data-export risk** | Direct MCP crosses Codex's approval gate, but a headless process has no interactive user to approve it | Approve the direct call in an interactive Codex session, or use the preferred background JSON CLI: `sparra role run … --json --out <file>` / `sparra eval … --json`. |
| **Codex cannot start the Sparra MCP server: `sparra-run-mcp` is missing / not found on `PATH`** | The plugin declaration intentionally launches the installed package bin; the repo clone alone does not put it on `PATH` | Run `npm link` from the Sparra checkout, confirm both `sparra` and `sparra-run-mcp` resolve, then reinstall/reload the plugin in a fresh thread. |
| **Decomposition is huge** (8–12 items for a small app; a standalone "scaffold" or "generate/build/verify" item) | Decomposer over-splitting — usually because it's running on Codex, which doesn't follow the decompose prompt as tightly as Claude | Set `roles.decomposer: { backend: claude, model: opus }`. Re-decompose (`sparra build --fresh` or clear `workitems/`). Decomposition is a *planning* act; keep it on Claude. |
| **Evaluator rejects the contract as "the wrong project" / quotes an unrelated plan** | The work dir is **nested inside another Sparra project**; a read-only role read the parent's `PLAN.md`/`.sparra/` up the tree | Run the project in its **own directory**, not nested. Remove stray parent `PLAN.md`/`.sparra/`. (Recent prompts anchor the evaluator on the in-message plan, but a clean dir is the real fix.) |
| **Contract never reaches AGREED / many rounds, run dies mid-`contracting`** | Either genuine, substantive critiques (healthy — it force-converges after `maxNegotiationRounds`) OR the session hit a turn cap | Read the last critique. If it's finding real holes, optionally raise `contract.maxNegotiationRounds`. If sessions are starving, ensure `build.maxTurnsPerSession` is generous (contract sessions use it). |
| **"verify-probe found N broken verify command(s) — re-opening negotiation"** | The harness probe dry-ran the agreed contract's "I will verify by" commands and hit a USAGE error (not found / unknown flag / usage text) or an UNSAFE command (safety-rule-rejected — chaining/redirect/network/mutation, an argv[0] outside the executor's build/test-runner allowlist, or a mutating SUBCOMMAND of an allowlisted runner — npm/yarn/pnpm/bun run only as `test` / `run <script>` / `run-script <script>`, and verbs like `npm version`, `cargo publish`, `go clean`, `mvn deploy` are denied; the harness can never run it) — the contract was bounced, not shipped broken | Working as intended. Read the `verify-probe (harness)` section in `.sparra/contracts/<id>.contract.md`. A legit-but-unknown tool? Declare it in `build.verifyCommands` (the explicit opt-in past the allowlist). If the probe misfires otherwise, set `contract.probeVerifyCommands: false`. |
| **"drop-guard found N uncited assertion drop(s) — re-opening negotiation"** / **"agreement voided by the drop-guard"** (a `### Round N — drop-guard (harness)` section in the contract file) | The harness diffed a round>1 revision's `## Assertions` against the previous proposal and found an assertion that **vanished without the answered critique naming it** — a lossy patch. A revision must PATCH (keep uncited assertions verbatim), so the drop bounced the contract back into negotiation (voiding a same-round AGREE), same pattern as the verify-probe | Working as intended — it stops critique-driven rewrites from silently dropping previously-agreed scope. Read the `drop-guard (harness)` section in `.sparra/contracts/<id>.contract.md`: the next round should RESTORE each listed assertion or JUSTIFY the drop by naming the authorizing critique point. A genuinely-cited drop shouldn't bounce (numeric ref / verbatim quote / significant-token overlap counts as cited); if the heuristic misfires it's a prompt/`sparra reflect` target. |
| **"pass demoted by the rerun gate" / item fails despite a passing verdict** | The contract's verify commands did not all stay runnable-and-green — mixed exits = FLAKY artifact, all-nonzero = failing-as-shipped, UNSAFE = safety-rule-rejected (never ran, so it can never be witnessed exiting 0; unknown tools AND mutating subcommands of allowlisted runners — `npm version patch`, `cargo publish` — need a `build.verifyCommands` opt-in) | Fix the flakiness/failure/unsafe command (the blocking feedback names the command + output) — rerun-to-green never passes. `build.flakinessReruns: 0` disables the gate. Set `build.flakinessLoadRerun: true` (off by default) to ALSO run each command once under concurrent CPU load, catching a suite that only times out under machine load. |
| **"PREFLIGHT bounce" — a round skips the evaluator and re-generates** (log names it, memory notes `preflight bounce`; feedback says "PREFLIGHT FAILURE — your OWN verify commands failed") | `build.preflightVerify` is on: after generation and BEFORE the evaluator, the harness ran the contract's own verify commands and one **deterministically failed** (ran, nonzero, not a *usage*/broken command) — so the round short-circuited back to the generator instead of spending an evaluator session. A *usage* (command-not-found) or *unsafe* (safety-rejected) outcome never bounces; only a real behavioral gate fail does | Working as intended — the generator's own gates must pass before an evaluator session is spent. Read the bounce feedback (it carries the failing command + output, holdout-redacted) and fix the failing gate. The bounce is **capped at one** before the evaluator must run, and it **advances the round** like a failed eval round. Set `build.preflightVerify: false` to disable the gate. |
| **Generator thrashes on the SAME assertion for several rounds — patches symptoms, never fixes it; round feedback now says "DIAGNOSE FIRST … state the ROOT CAUSE of #N"** | The per-assertion escalation register fired: that assertion's fail streak reached `build.assertionEscalateAfter` (K, default 2), so the harness UNCAPPED its evidence and prepended a diagnose-first instruction — a register between a plain patch and a GAN pivot | Working as intended — read that assertion's now-**uncapped** evidence in the latest verdict; the root cause is usually visible in the error-bearing tail that a capped view was hiding. If the generator still can't converge after escalation, it's a candidate for **quality escalation** (`build.escalateAfterRounds` + a stronger `escalation` role) or a GAN pivot. Lower K to escalate sooner, or set `build.assertionEscalateAfter: 0` to disable. |
| **Clean, working artifact scores just under threshold / fails on a weird assertion** | Over-specified or impossible contract assertion (toolchain trivia, "prove not-X", environment-controlled property) | Read the contract's assertions and the failed one's evidence in the verdict. Proportionality is enforced in the prompts; if it slipped through, it's a prompt-tuning target (`sparra reflect`). |
| **Item passed, but `reconcile` then marks a plan behavior "pending / not satisfied"** | Gamed/degenerate pass: an assertion met by a no-op input (a combine step fed identical copies, a structurally-wrong fixture, a stub, or a literal term reinterpreted loosely) — the core behavior never ran on real data | Read the verdict's assertion evidence and the contract. The anti-gaming guards (distinct fixtures, contrasting negative case, structurally-correct stand-ins) live in the contract-generator/contract-evaluator/evaluator prompts; if it slipped through, `sparra reflect` and harden them. |
| **Item passed, but its own committed tests / contracted verify commands crash as shipped** (wrong flag, bad field, import error) | Evaluator laundered a broken verification harness: it hand-corrected the invocation to get a green instead of failing the as-shipped path. Often the bad command survived contract negotiation and got copied into the committed tests | Read the verdict — does it note running a hand-substituted command? The fix lives in the prompts: contract-(generator/evaluator) require every verify command to be confirmed runnable against the real CLI; the evaluator must run shipped verification as-is and not launder it. `sparra reflect` if it recurs. |
| **Codex evaluator never runs the tests / `EPERM` writing `node_modules/.vite-temp` (or any in-repo scratch); eval silently degrades to code-review-only** | The Codex evaluator's OS sandbox was `read-only`, so test/build tools can't write the scratch they need | Default is now `exercise.sandbox: workspace-write` — the exercising evaluator gets writable scratch (network stays off; a source-integrity guard reverts+fails any artifact-source write) **on a worktree/branch boundary**. If it's still read-only, you're on an in-place run (no branch) or set `exercise.sandbox: read-only`. Build on a worktree (`git.strategy: worktree`). **For an interactive `run_role`/`sparra-loop` eval, pass `worktree: true`** (MCP `run_role`) / `--worktree` (`sparra eval`) so the eval runs in a temp WIP-snapshot worktree with writable scratch — an in-place `run_role` eval is read-only and false-blocks here (the MCP tool silently dropped `worktree` before it was wired through — fixed). The **Claude** evaluator exercises via the in-process runner and was never affected. When the exercise genuinely *still* can't run, the evaluator marks the verdict `exerciseStatus: blocked` (**inconclusive**) — the loop no longer treats that as a behavioral fail or GAN-pivots on it; it's surfaced for you as needs-attention rather than failing correct work. |
| **`run_role`/`sparra eval` aborts before launch: "expectedHead mismatch … is at HEAD X but the brief cites Y"** | Eval-provenance guard: the tree the judge would grade (the source checkout's HEAD on a `worktree` run, the workspace HEAD in place) isn't the commit the brief cites — the run is refused **before any tokens are spent** so a judge never grades the wrong tree | Working as intended. Check out the cited commit (or fix the SHA in `expectedHead`/`--expected-head`) and re-run. On a `worktree` run the graded workspace is a **detached WIP-snapshot commit whose parent is the verified HEAD**, so a differing `git rev-parse HEAD` *inside* the workspace is expected, not tampering. A bad `evalBaseRef`/`--eval-base` aborts the same way, naming the ref. Both controls are judge-only (rejected on a writer/contract-generator). |
| **Evaluator prompt contains `[VERIFIED BASELINE @ <sha>]` or `[VERIFIED BASELINE: UNAVAILABLE …]`** | The verified-baseline opt-in (`baselineCommand` + `evalBaseRef`) was active. **`[VERIFIED BASELINE @ <sha>]`** — the RUNNER ran the allowlisted command at the base-ref SHA in a throwaway worktree before spending any tokens; the block is authoritative: failures listed there are runner-verified pre-existing; any failure NOT in the baseline is a NEW regression. The generator's prose carveout for "pre-existing failures" carries no authority alongside this block. **`[VERIFIED BASELINE: UNAVAILABLE — <reason>]`** — the baseline infrastructure failed (worktree-add, dep provisioning, or spawn error) AFTER the base ref resolved; the evaluator fell back to grading with no verified baseline, and the note explicitly states that prose alone is NOT a verified pre-existing-failure manifest. The eval proceeds — it is not aborted. | For `UNAVAILABLE`: read `<reason>` (worktree-add path conflict, provision error, etc.) and fix the environment, or omit `--baseline-command` for an unverified eval. For a bad `evalBaseRef` (unresolvable), the run aborts pre-launch (see row above). To add the feature to a `/sparra-loop` eval: pass `baselineCommand=<cmd>` (MCP `run_role`) / `--baseline-command <cmd>` (`sparra eval`/`role run`), with `evalBaseRef` set. The command must prefix-match a `build.verifyCommands` entry; chains/pipes/redirects are rejected. |
| **Verdict lists `unrunAssertionIds` / `exerciseStatus: mixed` after missing tool, dead Simulator/CoreSimulator, sandbox kill, or command-not-found** | The evaluator environment could not execute one or more gates. This is narrower than a behavioral failure: the gate produced no artifact signal, while other gates may have run (`mixed`) | Treat UN-RUN ids as **no signal**, not failed assertions. Read the verdict's un-run section and `notes` for env evidence; fix the evaluator environment/toolchain or accept that the conductor's already-verified gate is authoritative. A `mixed` verdict is still gradeable from observed gates; an all-UN-RUN verdict is inconclusive and does not advance failed-round or pivot streaks. |
| **Verdict fails with "Unobserved pass: no mcp__exercise__ activity backed this pass"** (pass demoted because the exercise was not observed) | Observed-run gate (`exercise.requireObservedRun`, default on): the evaluator reported a PASS but routed **zero** commands through `mcp__exercise__run_command`/`http_request`, so the harness saw nothing backing it — on mechanisms `cli`/`web` an unobserved pass is pure self-report and is demoted to fail | Working as intended — on an **in-process-MCP** eval backend (Claude) the evaluator must exercise via **`run_command`** (not raw Bash) so the harness observes real exit codes; the blocking note feeds that back and the next round should re-run gating commands through the tool. `ios`/`computer-use`/`custom` are exempt. **Backend caveat:** on a backend WITHOUT in-process MCP (Codex — `BackendCapabilities.inProcessMcp === false`), the `mcp__exercise__*` tools are never attached, so the gate does **not** fire — that evaluator exercises via its own **native command runner** and self-reports `exerciseStatus` honestly (the harness can't classify exit codes there), and zero `mcp__exercise__` activity is expected, not a demotion. If a `cli`/`web` project genuinely can't route its exercise through the tools, set `exercise.requireObservedRun: false`. |
| **Verdict fails with "Integrity violation: the evaluator wrote N artifact file(s) during exercise (reverted)"** | The evaluator modified the artifact source it was grading (e.g. "fixed" the code to pass) while exercising under `workspace-write` — the source-integrity guard reverted those writes and failed the verdict | Working as intended: the verdict can't be trusted when the grader edited the code. Read the named files; the evaluator prompt should exercise, not edit. If it recurs, `sparra reflect` to harden the evaluator prompt. Gitignored scratch (caches, build temp) is never flagged — only the artifact surface. A **symlinked top-level `node_modules`** is canonicalized (`lstat`) and treated as scratch, so it is never flagged/deleted even though `.gitignore`'s dir-only `node_modules/` pattern lets git surface the symlink. |
| **Generator reports it couldn't run `tsc`/`npm test`, "verified by inspection" — ships uncompiled/untested code** | The writer's Bash wasn't auto-approved (permission mode doesn't auto-accept Bash), so it wrote blind | The generator now **self-verifies** on a worktree boundary: it auto-runs `build.verifyCommands` (typecheck/test/build) and fixes what it broke. An **interactive in-place** `run_role` (no worktree) used to be unable to self-verify — so the conductor had to run every gate out-of-band — but it now **can** when opted in: pass `allowVerify: true` (MCP `run_role`) / `--verify` (`sparra role run`) to enable the same strict allow-hook in place. If it still can't, check you're on a worktree (`git.strategy: worktree`) or passed the opt-in, and that the command starts with a `verifyCommands` entry and has no chaining/redirect/network/commit (those are disqualified). **Only the exact allow-listed forms are approved** — package-runner / path-qualified variants (`npx tsc`, `./node_modules/.bin/vitest`) are NOT, so the generator must run `npm run typecheck` / `npm test` etc. as written (it's told this). Add the project's command to `build.verifyCommands`. (Codex confines these to its sandbox; for Claude they run unsandboxed like the evaluator's exercise.) |
| **Claude contract-evaluator cannot prove a configured verify command** | The critique is running in place, without the isolated boundary required for Bash auto-approval | Run autonomous negotiation on its build worktree, or use `role run --kind contract-evaluator --worktree`. The Claude role then gets the shared strict `allowVerifyBash` hook for configured bare commands; unsafe/non-allowlisted forms remain blocked. Codex relies on its OS sandbox rather than hook guarantees. |
| **`run_role` result carries `verifyGateWarning` / evaluator grades the generator's assertions as "unverified"** | A generator role-run's contract references one or more configured `build.verifyCommands` (e.g. `npm test`) but self-verify was NOT enabled: no `allowVerify: true` / `--verify` and no `unitWorktree` / `build.branch` boundary — so those commands were approval-blocked and the generator could only claim them "unverified". The runner surfaces this at **launch time** as a `verifyGateWarning` field on the `run_role` result and in the phase log, naming the exact blocked commands | **Working as intended.** The advisory fires BEFORE the session so you can act: re-launch the generator with `allowVerify: true` (MCP `run_role`) / `--verify` (`sparra role run`) to enable the strict allow-hook in place, or use a `unitWorktree` / branch boundary where it's already enabled. The warning is holdout-safe (command strings only). If you see "unverified" assertions in the evaluator's verdict and there was no `verifyGateWarning`, check the role kind (the advisory fires for `generator` only) and confirm the commands appear verbatim in `build.verifyCommands`. |
| **iOS app letterboxed (320×480, black bars); UI taps "succeed" but miss** | Missing launch screen → iOS compatibility mode shrinks the logical screen | Add `INFOPLIST_KEY_UILaunchScreen_Generation: "YES"` (or `UILaunchScreen: {}`) to `project.yml`, `xcodegen generate`, rebuild. This is an **app defect**, not tooling. |
| **macOS app: evaluator can't screenshot/drive the UI; UI assertions fail "no evidence"** | A Mac app has no simulator and xcodebuildmcp's screenshot/ui-automation is simulator-only | Set `exercise.ios.platform: macos`. The UI is then verified via an **XCUITest** target (`macos test`) + `xcresulttool`-extracted screenshots + `screencapture`; ensure the generator built a UI-test target (it's told to on `platform: macos`). See `docs/ios.md` → macOS apps. |
| **iOS UI/animation graded "blind" — verdict passes on code/logs alone, or says "device-verify-pending", with no screenshot/animation evidence** | Visual correctness (layout/nav/**transition shape**) is a dimension code review can't see; the evaluator wasn't given the capture recipe | Keep **`exercise.ios.visual: true`** (default): the iOS guidance then carries the **screenshot** chain (Read the PNG + a11y-hierarchy) **and** the **animation** chain (`simctl io … recordVideo --codec=h264` → `ffmpeg … "fps=N,scale=W:-2,tile=CxR"` → Read ONE start→mid→end contact sheet). Ensure **ffmpeg** is installed (else the animation gate is **UN-RUN**, never a fail) and the generator exposes `#if DEBUG` launch-arg reach hooks. Honest boundary: captures prove geometry/nav/transition *shape*, **not** motion feel/jank/120 Hz/gesture/GPU-ML. See `docs/ios.md` → Visual verification. |
| **macOS/XcodeGen build never compiles — `xcodebuild` burns 100%+ CPU, no `swift-frontend`/`clang` workers, zero `.o` files, "hangs" for many minutes** | `project.yml` references a local SwiftPM package in the project's own directory (e.g. an engine package at the repo root) with `path: ./` — the trailing slash makes XcodeGen resolve it to the filesystem root `/`, emitting a folder reference, so Xcode recursively scans the whole disk on project load (stuck in `IDEContainer _locateFileReferencesRecursively`) and never starts compiling | Use `path: .` (no trailing slash) in `project.yml`, `xcodegen generate`, rebuild — builds in seconds. The generator is now told this; `sparra reflect` if it recurs. |
| **Budget warning says the USD cap did not bind / Codex item shows `costUsd` is $0** | The backend reports zero/unknown USD cost, so `maxBudgetUsdPerItem` can't stop the item | Set `build.maxTokensPerItem` for the explicit ceiling, or set `build.zeroCostTokenCap` as the fallback when the USD cap is active and `maxTokensPerItem` is off. |
| **An item tagged `gen: "local"` keeps failing / thrashing rounds** | Hybrid routing sent it to a local model (`roles.generatorLocal`, e.g. LM Studio) too weak for that item | Check the trace header for `endpoint: … (local)`. Remove the `gen` tag in `items.json` (→ main generator), or stop tagging by unsetting `roles.generatorLocal`. Keep local routing for trivial/mechanical items only. |
| **`gen: "local"` item ran on the main generator anyway (warning in log)** | `roles.generatorLocal` isn't configured, so the build fell back | Add `roles.generatorLocal: { backend: codex, model: <local-id>, baseUrl: http://localhost:1234/v1 }`, or drop the tag. |
| **An item PASSED the evaluator but wasn't accepted; feedback says "SECOND-OPINION … DISAGREED and FAILED it"** | The opt-in **second-opinion gate** (`evaluator.secondOpinion.enabled`) re-graded the primary PASS with `roles.evaluatorSecond` on a different backend/model, and that second opinion produced a real `fail` — so the accept was demoted (working as intended: a lenient primary can't launder slop) | Read the second opinion's blocking in the round feedback / verdict file and fix the genuine issue, or, if the disagreement is spurious, tighten the contract. Turn the gate off by unsetting `evaluator.secondOpinion.enabled`. |
| **`evaluator.secondOpinion.enabled` is on but the log says "second-opinion gate is a no-op"** | The independence guard fired: `roles.evaluatorSecond` is unset, or resolves to the **same** effective backend+model as the actually-selected primary evaluator (a same-model second opinion is pointless) | Set `roles.evaluatorSecond` to a **different backend/model** than the primary `roles.evaluator` (check whether the primary fell back to that same model — the comparison is post-fallback). |
| **`run_role` evaluator result carries `sameModelGrade: true`** (or the verdict header says `fell back from …` and the actual grader matches the generator) | The evaluator fell back from its configured backend (e.g. `codex/gpt-5.5`) to a fallback (e.g. `claude/opus`) that MATCHES the generator — the cross-model adversarial gate silently collapsed to a same-model grade. The `fallbackFrom` field names the originally-requested evaluator. This is only detected when the conductor passes `crossModelBaseline` (the generator's `backend`/`model`) on the evaluator call | Do NOT accept a `sameModelGrade: true` passing verdict as a genuine cross-model grade. Re-run the evaluator on a genuinely different backend once the primary limit clears (or configure a fallback on a third model family). The verdict header names the actual grader so you can confirm which model ran. |
| **`sparra conduct` run "hangs" with a unit non-final; `.sparra/conduct/<runId>/decisions/<seq>.request.json` exists but no `<seq>.decision.json`** | A **parked decision**: the decision engine surfaced a judgment point (`surface: park` / `park-timeout` before its `timeoutSec`) and is waiting for a human answer (file or TTY) | Answer it: `sparra conduct --decide <runId> <seq> <answer> [--note "…"]` (the answer must be one of the request's `options`), type it inline if a TTY prompt is open, or answer over the HTTP bridge (`POST /jobs/:id/decision {seq,answer}`). Under `park-timeout` it auto-resolves after `timeoutSec`; use `--auto` (or `conduct.decisions.surface: auto`) to never park. The request/decision are holdout-safe (built from `ParentSummary` only). |
| **A conduct decision record shows `source: "brain-fallback"` (or `auto-deterministic`)** | The LLM conductor **brain fell back**: `brain-fallback` = the brain was consulted but its JSON stayed invalid after the one reask, so the deterministic policy (the kind's default option) decided; `auto-deterministic` = no brain was available (e.g. `roles.conductor` unreachable / limit) so the deterministic default decided | Not a failure — the run still resolved deterministically and recorded the rationale. If you expected the brain to steer, check `roles.conductor` (backend/model reachable, not rate-limited) and the brain trace under the run dir; re-run with `--auto` off to park for a human instead of auto-deciding. |
| **`BUDGET_EXCEEDED` on an item** | Accumulated cost/tokens crossed the per-item cap | Expected — the run continues to the next item. Raise `maxBudgetUsdPerItem`/`maxTokensPerItem` if premature; check `memory.md` for the halt note. |
| **Build "hangs" mid-item with no output for a long time; `sparra status` says "paused on a provider limit — resumes ~HH:MM"** | `build.autoRestart` is on and a role hit a provider rate/usage window with no available fallback — it's *sleeping* until the window reopens, not stuck | Expected. Let it resume on its own, or re-run `sparra build` to retry now. State is on disk (`state.json`: `build.waitingUntil`, `build.limitedRoles`), so a kill loses nothing. Add a cross-provider `roles.<role>.fallback` to keep working instead of waiting. |
| **`BUILD PAUSED — provider limit`, phase stays `build`** | `build.autoRestart.maxRestarts` wait cycles exhausted — the run stopped cleanly, the in-flight item is left mid-build (not failed) | Re-run `sparra build` to resume from where it stopped (completed items skip). Raise `maxRestarts`/`maxWaitSec`, or configure a `fallback` model so it doesn't depend on waiting. |
| **A limit fires but the build keeps going on a different model** (log: "switching to a fallback model" / "generating with fallback …") | Working as intended: `roles.<role>.fallback` is set and the primary's backend is in a limit window | No action. It switches back once the primary's window reopens. If the fallback is also limited (same provider), it can't help — point it at a *different* backend. |
| **A session dies on `401 Unauthorized` / "missing bearer" / "not logged in · please run /login" / "invalid api key" / "authentication failed"** — but the item is NOT marked failed (round given back, log says it paused-and-retried / fell back) | An **auth/transport** failure: the backend session never ran, so it's classified as a `limitHit` (`kind: "auth"`), NOT a behavioral FAIL. `build.autoRestart` pauses-and-retries / falls back, exactly like a rate/usage limit — it never burns a round or pollutes calibration | Fix the credentials: re-auth the backend (`claude` / `codex login`, refresh the API key in `build.env`/`~/.codex`), then re-run `sparra build` to resume. Configure a cross-provider `roles.<role>.fallback` so an auth outage on one backend doesn't stall the run. |
| **Interactive `run_role`/`/sparra-loop`: a role returns nothing useful / a verdict scored 0 with "no parseable verdict", often on Codex mid-session** | The backend hit a provider limit OR returned a **silent empty completion** (an explicit `emptyCompletion` marker on the backend result) — classified as a `limitHit` when nothing ran, NOT a real result | Check `filesChanged` first: if the result instead carries `emptyCompletion: true` (writer + files changed), see the row below — the work landed. Otherwise `run_role` auto-falls-back to `roles.<role>.fallback`; if the whole chain is limited the result carries `limitHit` — switch that role to another backend (`--backend claude`) or retry later. Never feed a `limitHit` result back to the generator as a FAIL. Configure `roles.<role>.fallback` on a *different* backend to make this automatic. |
| **Interactive `run_role`/`/sparra-loop`: a generator returns `ok: false` with EMPTY result text, but `emptyCompletion: true` and `filesChanged > 0`** | The work fully LANDED on disk; only the completion report failed to emit (empty completion or a budget-cap death at the report boundary) | **Self-heals first:** with `build.jsonReask` on, a budget-cap death already got ONE automatic tightly-capped (1-turn) report-only re-ask on the same session — so a recovered report surfaces in `resultText` (an `errors` note records "recovered … via a one-shot re-ask") and `emptyCompletion` is cleared. If `emptyCompletion` STILL surfaces, the re-ask failed, was disabled, or it's a provider-limit empty completion (left to you). Then it's NOT a failure and NOT "re-run the item" — a second generator would clobber the landed work (the fallback chain already refuses to). RESUME via MCP with `resumeSessionId`/`resumeBackend`, or via CLI with `--resume-session <id> --resume-backend <backend>`, using the result's session/backend; alternatively verify and accept the landed work. |
| **`sparra role run --json` / `sparra eval --json` output is mixed with progress text, or a consumer still expects `result`** | The JSON surface should use the same holdout-safe builder as MCP: one object on stdout, human logs on stderr, and non-evaluator output under `resultText` | Ensure the caller reads stdout only and parses `resultText` plus `errors`; update stale consumers of `result`. Evaluator JSON correctly omits both `resultText` and `traceDir`—do not reconstruct either from traces. |
| **Interactive `run_role`/`/sparra-loop`: a result carries `hitBudget: true`** | The run stopped on OUR per-call USD cap (`maxBudgetUsd` / `build.maxBudgetUsdPerItem`) — not a provider limit, not a turn cap | Not a behavioral fail. If `filesChanged > 0` the work may already be done (see the `emptyCompletion` row). Resume the same session via `resumeSessionId` + a raised `maxBudgetUsd` to finish; never feed it back as a FAIL. |
| **Interactive `run_role`/`/sparra-loop`: a generator returns `noProgress: true` and changed no files** (often after burning tokens) | The writer changed no file's **content** (detection is content-based against a pre-run snapshot, so an edit to a file already dirty at run start on a continuation/fix round DOES count — this is no longer a false signal there): it had nothing actionable to do, or its reads/Bash were blocked. A role can ALWAYS read its in-scope workspace now (guard-level allow), so a genuine no-progress almost always means an empty/unactionable brief — not a permission wall | Don't feed it back as a behavioral FAIL. Check the brief actually asks for a file change and points at the right `workspace`; re-run with a sharper brief. If reads truly seem blocked, confirm the workspace/`additionalDirectories` cover the files (the in-scope read allow is keyed to those). |
| **Interactive `run_role`/`/sparra-loop`: a role returns `hitMaxTurns: true` with a partial artifact** | The role stopped at the per-session turn cap (`build.maxTurnsPerSession`), unfinished — not a failure | RESUME the same session: re-call `run_role` for the same role with `resumeSessionId` + `resumeBackend` = the result's `sessionId`/`backend` and a short "continue where you left off" brief (don't re-read the workspace, don't pivot). Raise `build.maxTurnsPerSession` if items routinely need more turns per session. **Report self-heals first:** if a generator hit the cap with work landed (`filesChanged > 0`) but no parseable report, `build.jsonReask` already fired ONE tightly-capped (1-turn, text-only) report-only re-ask on the same session — a recovered report rides in `resultText` (an `errors` note says "recovered … via a one-shot re-ask") while `hitMaxTurns` STAYS true (you still resume to finish the work). On the **autonomous build loop** a **Claude** generator ALSO gets a *preventive* one-time ~80%-of-`maxTurnsPerSession` turns-remaining warning nudging it to emit its report BEFORE the cap (Claude-only; see `docs/build-loop.md`), so forfeited-report cap deaths should be rarer there — the re-ask is the backstop. |
| **Item keeps failing the same criterion, then restarts from scratch** | GAN pivot: same rubric criterion below `pivot.threshold` for `pivot.N` rounds | Expected behavior. On the restart the workspace is reset to the item-start state (`pivot.resetWorkspace`, default true — only when `git.autoCommit` is on, the recorded branch carries `git.branchPrefix` (Sparra-owned; a recorded `main` refuses), and the workspace's live branch matches it; in-place runs never reset), and the fresh prompt carries a "PRIOR ATTEMPTS" ledger of what already failed (`state.json` → `build.items.<id>.attempts`). If it pivots forever, the contract or rubric expectations may be miscalibrated for the item. |
| **Build errored with `pivot workspace reset failed — halting rather than generating on a dirty tree`** | The pivot's workspace reset (`git restore`/`git clean -fd`) failed, so the fresh restart was refused | Deliberate: never generate on a dirty tree. Inspect the workspace's git state (lock files, permissions), fix or clean it manually, then re-run `sparra build`; or set `pivot.resetWorkspace: false` to pivot without resetting. |
| **A Glob denial mentions a protected target or unsafe root, or a role wastes turns moving every search under `src/`** | The holdout guard distinguishes an unsafe unfiltered root from a pattern whose resolved matches target protected artifacts | Keep broad searches under a relative subtree such as `src/`. A selective repository-wide filename pattern such as `**/vitest.config.*` is allowed; broad Markdown/any-file patterns and protected-basename alternatives remain denied. |
| **Build output / screenshots litter the project root** | Older guidance; the evaluator should use a temp derivedDataPath + scratch dir | Current guidance handles this; if seen, it's a prompt-tuning target. |
| **Evaluator/contract-evaluator verify probe dies at startup on `EPERM … node_modules/.vite-temp/vitest.config.ts.timestamp-…` (or a `/var/folders` mkdir)** | Vitest writes temp files under `node_modules/.vite-temp` + the OS temp dir; a read-only judge sandbox EPERMs them *before collection*, so the probe never runs (a pure-environment FAIL, not an artifact defect) | Handled: both judge roles now get the **default writable-scratch env layer** (`TMPDIR` redirected into a per-run `sprj-<hex>` dir; `src/build/judgeScratch.ts`). The in-repo `node_modules/.vite-temp` write is covered by the workspace-write scratch carve-out on an **isolated checkout** (build branch / `--worktree`). If still seen on an in-place run, use `sparra eval --worktree` (or set `exercise.sandbox`) so the exercise gets writable scratch. |
| **`node bin/sparra.mjs …` / `npm run sparra -- …` smoke dies on `EPERM` creating a socket under `tmpdir/tsx-<uid>/<pid>.pipe` before any Sparra code runs** | The packaged CLI shells to **tsx**, whose IPC pipe path is derived from `os.tmpdir()` (installed tsx 4.x: `join(os.tmpdir(), 'tsx-<uid>')` then `join(tmpdir, '<pid>.pipe')`); a read-only sandbox can't create it | The judge env layer redirects `TMPDIR` into the writable per-run scratch dir, which fixes the pipe PATH's **writability** — but that alone does **NOT** make the tsx socket usable under a sandboxed judge: the sandbox POLICY still denies `listen(2)` on the Unix-domain socket (next row), so a tsx-launched smoke that IPCs over the `.pipe` **UN-RUNs** regardless. Under a **Claude** (no-OS-sandbox) judge the redirect is sufficient. Under Codex, mark the smoke **UN-RUN** (capability-probe skip). For an ad-hoc smoke, run outside the sandbox (or point `TMPDIR` at a writable dir on a backend without the policy deny). |
| **A gate dies on `listen EPERM` / `bind EPERM` on a `.pipe` or socket path *even with a writable TMPDIR*** (a raw `net.createServer().listen()` probe reproduces it; tsx CLI smokes, dev servers, any socket LISTENER) | The Codex judge sandbox denies **Unix-domain-socket LISTEN as seatbelt POLICY**, independent of path writability — proved twice with a raw `net.createServer().listen()` probe inside a writable scratch TMPDIR. ## Clean re-runs

Cached state is reused on purpose (resumability), which can mask a fix. To force a genuinely
fresh build:

- **Re-decompose + re-contract**: clear the cached pieces, then `sparra build --fresh`
  (resets items/decomposition for the run). To also pick up updated **prompts**, remove the
  seeded copies so they fall back to the latest defaults:
  ```bash
  rm -rf .sparra/contracts .sparra/workitems .sparra/verdicts .sparra/prompts
  sparra build --fresh
  ```
- **Total reset** of a throwaway project: `rm -rf .sparra <generated-files>` then
  `sparra init && (cp your-config .sparra/config.yaml) && sparra freeze && sparra build`.
- Prompt/config changes only reach an **already-initialized** project on re-seed (delete
  `.sparra/prompts/`) or re-init; the example `run.sh` does a fresh `init` each time.

## Watching a live run

```bash
ls -lt .sparra/traces/*/ | head          # newest trace = current activity
tail -f .sparra/traces/<run>/NN-<role>.md # follow a role as it works
```
The role filename tells you the phase: `decomposer`, `contract-generator`/`contract-evaluator`,
`generator-<id>`, `evaluator-<id>-r<n>`, `reviewer-<id>-r<n>`, `reconcile-<id>`, `reflector`.
